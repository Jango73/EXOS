# Realtek RTL8111 / RTL8168 / RTL8411 Driver Plan

This plan defines a single EXOS driver for the common Realtek PCIe Gigabit Ethernet family built around `RTL8111`, `RTL8168`, and `RTL8411` controllers.

The primary target is the Predator on-board controller:
- [ ] `10ec:8168` must be detected and operational

The implementation must avoid a one-off Predator-only path. The goal is one shared driver for the largest realistic set of Realtek PCIe Gigabit Ethernet controllers that expose the same broad model:
- [ ] `RTL8111`
- [ ] `RTL8168`
- [ ] `RTL8411`

This family grouping is the right first target because:
- the Predator device is already inside it (`10ec:8168`)
- these controllers are common desktop and laptop PCIe Ethernet devices
- they fit the existing EXOS Ethernet model much better than Intel Wi-Fi or USB network hardware

## Scope

### In scope
- [ ] PCI detection and attach for the Realtek PCIe Gigabit Ethernet family
- [ ] MMIO register access
- [ ] EEPROM or register-based MAC retrieval
- [ ] RX and TX descriptor rings
- [ ] link-state reporting through `DF_NT_GETINFO`
- [ ] polling mode
- [ ] legacy INTx interrupt mode
- [ ] integration with `NetworkManager`
- [ ] bring-up on Predator `10ec:8168`

### Out of scope for the first milestone
- [ ] MSI or MSI-X
- [ ] Wake-on-LAN
- [ ] power-management tuning
- [ ] jumbo frames
- [ ] checksum offload
- [ ] TCP segmentation offload
- [ ] multiple hardware revision tuning tables beyond what is required for stable bring-up

## Driver family strategy

### Step 0 - Define the supported family
- [x] create `kernel/include/drivers/network/RTL8169.h`
- [x] create `kernel/source/drivers/network/RTL8169.c`
- [x] use one EXOS driver name for the whole shared family, not per-device files
- [x] model the supported PCI IDs as a table of matches, not hardcoded single-device checks

Recommended naming:
- [x] use `RTL8169` as the source file family name if the driver is intended to cover the broad Realtek PCIe Gigabit Ethernet register family

Rationale:
- Linux and BSD drivers typically group these controllers into one large `r8169`-style family
- EXOS should follow the same hardware grouping logic instead of making `RTL8168-Predator-only`

### Step 1 - Start with the minimum safe PCI ID set
- [x] add `10ec:8168` first
- [x] add the common `RTL8111` and `RTL8411` identifiers that share the same base register model
- [x] keep the match table data-driven so new IDs are added without restructuring the driver

## Implementation steps

### Step 2 - Clone the E1000 integration shape, not the Intel-specific logic
- [ ] copy the structural integration points from `E1000.c` only as a reference
- [ ] keep the same driver contract: `DF_NT_RESET`, `DF_NT_GETINFO`, `DF_NT_SEND`, `DF_NT_POLL`, `DF_NT_SETRXCB`
- [ ] register the driver through `KernelData.c`
- [ ] keep `NetworkManager` unchanged except for normal attach and initialization flow

Acceptance:
- [ ] the Realtek device appears in `GetPCIDeviceList()`
- [ ] `SystemDataView` shows the Realtek controller as kernel-attached on Predator
- [ ] the attached driver type is `DRIVER_TYPE_NETWORK`
- [ ] `NetworkManager_FindNetworkDevices()` picks it up without special-case code

### Step 3 - Build a reusable hardware-description layer inside the driver
- [ ] define a compact per-device info table for revision-specific quirks
- [ ] keep register offsets, descriptor formats, and reset flags in one shared header
- [ ] separate common ring logic from revision-specific setup
- [ ] avoid scattering magic values through probe, reset, TX, and RX paths

Acceptance:
- [ ] adding a new Realtek PCI ID is done by extending a table, not by duplicating functions

### Step 4 - Probe and MMIO mapping
- [ ] validate `PCI_CLASS_NETWORK` and `PCI_SUBCLASS_ETHERNET`
- [ ] enable bus mastering and memory space
- [ ] decode and map the MMIO BAR
- [ ] validate that the mapped register block is readable
- [ ] read hardware revision information early and store it in the device context

Acceptance:
- [ ] driver logs show stable probe, mapped BAR, and revision identification

### Step 5 - Reset and baseline device initialization
- [ ] implement software reset with timeout handling
- [ ] wait for the controller to leave reset cleanly
- [ ] apply the minimum required initialization sequence for RX and TX enablement
- [ ] leave advanced offloads disabled for the first version
- [ ] keep link configuration conservative and stable

Acceptance:
- [ ] controller reaches a quiet idle state after probe
- [ ] repeated boot does not hang in reset or leave the device wedged

### Step 6 - MAC address and link reporting
- [ ] read the permanent MAC address from the correct register or EEPROM path
- [ ] populate `NETWORK_INFO.MAC`
- [ ] report `LinkUp`, `SpeedMbps`, `DuplexFull`, and `MTU`
- [ ] use `1500` MTU in the first implementation unless hardware setup requires another default

Acceptance:
- [ ] `network devices` and `SystemDataView` show a valid MAC address
- [ ] link up or down state matches the actual cable state

### Step 7 - RX and TX rings
- [ ] define Realtek descriptor structures in the driver header
- [ ] allocate DMA-visible RX and TX rings
- [ ] allocate packet buffers
- [ ] program ring base, count, head, and tail registers
- [ ] implement one-frame TX path first
- [ ] implement RX polling first, even before interrupts

Acceptance:
- [ ] a raw Ethernet frame can be transmitted
- [ ] received frames reach the registered RX callback in polling mode

### Step 8 - Polling-only network bring-up
- [ ] implement `DF_NT_POLL`
- [ ] make `NetworkManager` operate on the Realtek device in polling mode first
- [ ] validate ARP, IPv4, DHCP, UDP, and TCP without interrupt dependency

Acceptance:
- [ ] `sys_info` or `network devices` reports initialized state
- [ ] DHCP succeeds on a working network
- [ ] `netget` can download over the Realtek controller in polling mode

### Step 9 - Legacy interrupt support
- [ ] add INTx support through `DeviceInterruptRegister`
- [ ] implement a small top half
- [ ] move RX processing to deferred work or poll-style bottom-half handling, following the E1000 pattern
- [ ] keep interrupt acknowledgement and re-arm logic explicit and traceable

Acceptance:
- [ ] RX works with interrupts enabled
- [ ] no interrupt storm appears in shared-IRQ situations

### Step 10 - Revision handling for more Realtek variants
- [ ] identify the smallest set of revision quirks needed for stable support across `RTL8111/8168/8411`
- [ ] store those quirks in a data table
- [ ] keep the common fast path shared
- [ ] only split initialization stages when real hardware differences require it

Acceptance:
- [ ] Predator `10ec:8168` works
- [ ] at least one additional board from the same family works without code duplication

## Required architecture rules

### No local hacks
- [ ] do not add Predator-specific code paths
- [ ] do not special-case one PCI ID outside the match and quirk tables

### Reuse shared kernel patterns
- [ ] reuse `DeviceInterrupt` for interrupt routing
- [ ] reuse `DeferredWork` for bottom-half processing
- [ ] reuse the existing `NetworkManager` attach and maintenance flow
- [ ] if a generic DMA ring helper becomes necessary for multiple drivers, place it in `kernel/include/utils` and `kernel/source/utils`

### Logging rules
- [ ] all logs must keep the `[FunctionName]` prefix
- [ ] warnings and errors must stay short and human-readable
- [ ] raw register dumps must stay under `DEBUG()`

## Validation checklist

### Bring-up validation
- [ ] PCI detection succeeds on Predator
- [ ] driver attaches automatically
- [ ] MAC address is stable across boots
- [ ] link state changes when cable is plugged or unplugged

### Network validation
- [ ] ARP request and reply flow works
- [ ] DHCP lease acquisition works
- [ ] IPv4 send and receive works
- [ ] UDP socket send and receive works
- [ ] TCP connection establishment works
- [ ] `system/netget` works on the Realtek controller

### Stability validation
- [ ] no boot hang when cable is absent
- [ ] no boot hang when cable is present
- [ ] no interrupt flood after link changes
- [ ] repeated reboot does not leave the controller in a broken state

## Recommended delivery order

1. [ ] add the new driver skeleton and PCI ID table
2. [ ] implement probe, reset, MMIO mapping, and MAC read
3. [ ] implement TX and RX polling
4. [ ] validate DHCP and `netget` on Predator
5. [ ] add INTx interrupt mode
6. [ ] extend the PCI ID and quirk tables for more `RTL8111/8168/8411` variants

## Completion criteria

- [ ] Predator `10ec:8168` is fully usable as the primary EXOS network device
- [ ] the driver covers a shared Realtek PCIe Gigabit Ethernet family, not one machine
- [ ] the driver integrates with the existing EXOS network stack without local architecture exceptions
